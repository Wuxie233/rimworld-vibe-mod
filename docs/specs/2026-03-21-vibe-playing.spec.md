# Spec: RimWorld Vibe Playing — AI 辅助殖民地管理 Mod

## 做什么

- 构建一个 RimWorld C# mod，允许玩家通过自然语言表达高层战略意图（如"我要扩建防御工事"、"优先发展研究"），由外部 LLM 分析当前殖民地状态，生成具体执行方案并（经用户确认后）自动执行
- 实现殖民地状态的结构化提取与序列化，包括：殖民者属性/心情/技能、资源库存、建筑布局、威胁态势、研究进度、电力/食物状况
- 提供游戏内 UI Tab，展示 AI 的态势分析报告、方案建议、当前殖民地不足/风险点

## 不做什么

- **不做全自动无人值守玩法** — 战略决策权始终在玩家手中，AI 只负责执行层
- **不做训练/微调模型** — 使用通用 LLM API（OpenAI / Claude / Ollama 等），通过 prompt engineering 实现
- **不做实时战斗微操** — 战斗中的即时操控不在首版范围内（tick 延迟不可接受）
- **不修改原版游戏核心逻辑** — 不使用 Harmony Transpiler 修改原版方法，只做 Postfix/Prefix 监听 + 原生 API 调用
- **不做多人联机支持**
- **不做 Steam Workshop 自动发布流程**（首版手动管理）

## 边界条件

- **Token 预算**: 单次 LLM 调用的殖民地状态 context 不超过 8K token（约 6000 字），需要分层摘要策略
- **API 频率**: 默认每游戏内 1 天（~60 秒真实时间 @ 1x 速度）主动分析一次；用户手动请求不限
- **游戏版本**: 目标 RimWorld 1.5+（兼容 1.6）
- **延迟容忍**: AI 分析期间游戏可正常运行（异步 HTTP），方案执行采用确认后批量下达
- **存档安全**: mod 添加/移除不应导致存档损坏（所有自定义数据存储在 mod 自有 MapComponent 中）
- **大型殖民地**: 50+ 殖民者、200+ 建筑的地图仍需在 Token 预算内完成状态提取

## 错误处理

| 场景 | 处理方式 |
|---|---|
| LLM API 请求超时/失败 | 在 UI 显示错误提示，保持上次分析结果，不影响游戏运行 |
| LLM 返回无法解析的指令 | 记录日志，跳过该指令，向用户展示原始响应供人工解读 |
| 执行指令时目标无效（如资源不足、位置被占） | 中止该条指令，报告失败原因，继续执行后续指令 |
| API Key 未配置 / 无效 | Mod Settings 页面提示配置，禁用 AI 分析功能但不崩溃 |
| 殖民地状态序列化异常 | 捕获异常，输出到 RimWorld log，回退到上次有效状态快照 |

## 验收标准

### Phase 1: 只读分析（MVP）

- [ ] Mod 可在 RimWorld 1.5+ 正常加载，无红色错误
- [ ] 在 Mod Settings 中配置 LLM API endpoint + key 后，游戏内 Tab 可展示殖民地状态摘要
- [ ] 点击"分析"按钮后，AI 返回当前殖民地的态势分析文字（优势/劣势/风险/建议），在 Tab 中展示
- [ ] 状态序列化生成的 JSON 不超过 8K token
- [ ] 添加/移除 mod 不损坏存档

### Phase 2: 用户确认执行

- [ ] AI 返回的建议包含可执行 action list（JSON Schema 格式）
- [ ] 用户可逐条确认/拒绝 AI 建议的操作
- [ ] 确认后的操作（如设置工作优先级、指定区域、放置建筑蓝图、设定狩猎/伐木指标）能正确执行
- [ ] 执行结果在 UI 中有成功/失败反馈

### Phase 3: 周期自动化

- [ ] 在用户授权的策略范围内（如"自动管理食物和木材储备"），AI 可周期性检查并自动下达指令
- [ ] 用户可随时暂停/恢复自动化
- [ ] 自动模式下每个执行动作有日志可追溯
